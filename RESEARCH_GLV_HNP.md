# GLV-Aware Hidden-Number Problem on secp256k1 — A Research Proposal

A focused research-programme launch document.  Proposes a **novel
variant** of the Hidden-Number Problem (HNP) tailored to GLV-
endomorphism-using ECDSA implementations on j=0 curves, with secp256k1
the primary target.

> **Status**: research proposal.  No empirical validation yet — this
> is the "Phase 1" launch document.  The proposal needs (i) a concrete
> lattice construction with predicted dimension and (ii) a falsifier
> implementation before any claim of attack viability.
>
> Companion to [`RESEARCH_HNP_LANDSCAPE.md`](RESEARCH_HNP_LANDSCAPE.md)
> which surveys the broader HNP landscape.

## 1. Motivation

Modern secp256k1 implementations universally use the GLV (Gallant-
Lambert-Vanstone) endomorphism for fast scalar multiplication.  When
signing, the nonce `k` is decomposed as

```
        k  ≡  k_1  +  k_2 · λ    (mod n)
```

where `λ` is the eigenvalue of the order-3 endomorphism `ω` acting on
`E(F_p)[n]`, and `(k_1, k_2)` are 128-bit half-scalars.  Scalar
multiplication `k·G` then becomes `k_1 · G + k_2 · (ω·G)` with two
parallel 128-bit windowed multiplications — roughly 2× faster than a
single 256-bit scalar multiplication.

**Side-channel implication**: an implementation that processes
`k_1` and `k_2` in separate code paths can leak side-channel
information about *each half* of the nonce, rather than the
unified `k`.  Existing HNP attacks (Boneh-Venkatesan, Howgrave-Graham-
Smart, BBSP) are tailored to **biased k** — they assume an attacker
learns bits of `k`.  No published HNP variant (to the knowledge of
this research) is tailored to **biased k_1 OR k_2 separately**.

The natural research question: **does GLV-aware leakage produce a
different lattice attack profile than standard HNP, and if so, what
does the attack-cost vs. leak-bits curve look like?**

## 2. Setup: ECDSA signing with GLV decomposition

ECDSA signing on secp256k1:

```
Algorithm ECDSA_Sign_GLV(secret d, message hash h):
    Sample fresh nonce k ∈ [1, n−1]
    Decompose k = k_1 + k_2 · λ  (mod n) using GLV
    Compute R = (k_1 · G + k_2 · (ω·G)).x  ≡ k · G .x  (mod n)
    Set r = R.x mod n
    Compute s = k⁻¹ (h + d r) mod n
    Output (r, s)
```

The decomposition `k → (k_1, k_2)` is deterministic for a given `k`:

```
        k_1  =  k − k_2 · λ
        k_2  =  ⌊k · μ / n⌉   (rounded)
```

for a specific lattice basis `(λ, μ)` with `|k_1|, |k_2| ≈ √n`.
Without loss of generality, assume `k_1, k_2 ∈ [0, 2^128)`.

### Threat model: GLV-aware side-channel

We assume the attacker observes a side-channel that leaks **partial
information about `k_1` and `k_2` separately**.  Concrete scenarios:

- **Cache timing**: per-half table lookups in the windowed scalar
  multiplication leak (k_1 mod w, k_2 mod w) for window size w.
- **Branch prediction**: the parity or top-bit of each half-scalar
  affects which branch is taken in conditional subtractions.
- **Power analysis**: the Hamming weight of (k_1, k_2) separately is
  observable in low-noise environments.

The "**1-bit-per-half**" leakage model is the simplest: attacker
learns top bit (or some other fixed bit) of `k_1` and top bit of
`k_2`, per signature.  Standard HNP literature treats this as
"unstructured noise on `k`" — but the structure is GLV-specific
and a custom lattice should exploit it.

## 3. The proposed HNP variant

### 3.1 Signature equation in GLV form

Starting from the ECDSA equation:

```
        k  ≡  s⁻¹ (h + d · r)   (mod n)
```

Substituting `k = k_1 + k_2 · λ`:

```
        k_1  +  k_2 · λ  ≡  s⁻¹ · h  +  s⁻¹ · r · d   (mod n)
```

Define `A = s⁻¹ · h mod n` and `B = s⁻¹ · r mod n`.  Each signature
gives:

```
        k_{i,1}  +  λ · k_{i,2}  ≡  A_i  +  B_i · d   (mod n)
```

The secret `d` is the only unknown shared across all signatures.
Both `k_{i,1}` and `k_{i,2}` are unknowns per signature.

### 3.2 The "small-2D" lattice

Suppose the attacker knows that `k_{i,1} ∈ [0, 2^{127})` (top bit is
0) and `k_{i,2} ∈ [0, 2^{127})` (top bit is 0).  Define the small
quantity:

```
        x_i  :=  k_{i,1} + λ · k_{i,2}  mod n     ∈ {0, …, n−1}
```

For random `k_{i,1}, k_{i,2}` in `[0, 2^{127})`, `x_i` is "concentrated"
in a 2-dimensional sub-region of `(Z/n)` of size `≈ 2^{254}`, with
the specific shape

```
        x_i  =  k_{i,1}  +  λ · k_{i,2}  −  q · n   for some q
```

where `q ∈ {0, 1}` depending on the carry.

### 3.3 Lattice construction (sketch)

For `m` signatures, build the lattice `L ⊂ Z^{m+1}` generated by the
columns of

```
       ┌                                                  ┐
       │  n      0      0     …    0    0    │
       │  0      n      0     …    0    0    │
       │              ⋱                       │
       │  0      0     0      …    n    0    │
       │  B_1   B_2   B_3    …   B_m   1     │
       └                                                  ┘
```

— this is the **standard HNP lattice on `d`**.  The Boneh-Venkatesan
target vector is

```
        t  =  (A_1, A_2, …, A_m, 0)
```

and the close-vector point gives

```
        v  =  (A_1 + B_1·d,  A_2 + B_2·d,  …,  A_m + B_m·d,  d)
            =  (k_{1,1} + λ·k_{1,2},  …,  k_{m,1} + λ·k_{m,2},  d)   (mod n)
```

Each entry is `x_i` (mod n).

**The novelty**: standard HNP treats each `x_i` as "small in absolute
value mod n" (the residue's distance from 0).  In our setting, `x_i`
is "small in a 2D structured way" — confined to a specific
parallelogram with side lengths `2^{127}` and `λ · 2^{127}`.

**The GLV-aware claim**: by augmenting `L` with **two basis vectors
per signature** — one for the `k_{i,1}` half, one for `λ · k_{i,2}`
— the lattice problem becomes a 2m+1-dimensional CVP instead of m+1.
The shorter close-vector lives in a subspace constrained by both
halves, giving (heuristically) tighter recovery for the same total
leak-bits.

### 3.4 Predicted complexity

The information-theoretic argument:

- Standard HNP with `b` leaked bits per signature needs `≈ log₂(n)/b
  + constant` signatures to recover `d`.
- For `b = 1`: ~256 signatures, lattice dimension ~257.

In the GLV-aware regime:

- Each signature leaks `2` separate "1-bit constraints" — one on
  `k_{i,1}`, one on `k_{i,2}`.  Total leak: `2` bits per signature.
- Naive expectation: `~128` signatures needed.
- But the lattice is `2x` larger (2m+1 dim instead of m+1), so the
  LLL reduction cost scales as `dimension^5` to `dimension^6`.
- Net: roughly comparable wall-clock cost to standard HNP at the
  same total leak-bits.

**Where the GLV-aware approach should win**: in attack models where
**only the half-scalar `k_1` leaks** (not `k_2` or vice versa), or
where the **leak shape** is GLV-specific (e.g., `k_{i,1} = k_{i,2}`,
which sometimes happens in buggy implementations).

In the **k_1-only** model:
- Standard HNP gets noise on `k = k_1 + λ·k_2`, with the `λ·k_2`
  term acting as ~256-bit *uncontrolled* noise.  HNP fails.
- GLV-aware HNP isolates the `k_1` constraint and treats `k_2` as a
  free variable per signature.  Recovers `d` with ~256 sigs at
  1 bit/sig, lattice dimension ~2m+1.

This regime — "leak on one half, not the other" — is the
**proposed attack's natural niche** and the place to focus the
falsifier.

## 4. Falsifier programme

### Phase 1.5: working LLL Boneh-Venkatesan demo (status: ✓ done)

The codebase's `crypto/src/cryptanalysis/hnp_ecdsa.rs` already
implements the full Boneh-Venkatesan lattice attack with LLL and
BKZ reduction.  An integration test in
`crypto/tests/curve_audit.rs`
(`hnp_recovers_p256_key_via_lll_phase15`) confirms it recovers a
planted P-256 key from 8 biased signatures with `k_bits = 192`
(64 bits of nonce bias) in seconds.

**Empirical finding (LLL-degeneracy phenomenon)**: the same
parameters do *not* converge on secp256k1 lattices.  A controlled
head-to-head probe in
[`tests/lll_degeneracy_probe.rs`](tests/lll_degeneracy_probe.rs)
measured the following with `k_bits = 192`, `m = 8`, identical
seed pairs across both curves:

| seed pair                      | P-256 LLL      | secp256k1 LLL                    |
|--------------------------------|----------------|----------------------------------|
| (0xC0FFEE, 0xC0FFEE)           | ✓ 690 ms       | ✗ iteration cap @ 9993 ms        |
| (0xDEADBEEF, 0xBADCAFE)        | ✓ 697 ms       | ✗ iteration cap @ 9845 ms        |
| (0x12345678, 0x9ABCDEF0)       | ✓ 646 ms       | ✗ iteration cap @ 10119 ms       |

**P-256: 3/3 recoveries.  secp256k1: 0/3 recoveries.**

This is a *systematic* curve-specific effect, not seed-dependent
noise.  Wider seed sweeps (6+ deterministic + BKZ-10) reproduce
the same pattern.

**Cross-Koblitz refinement (RUN — original hypothesis REFUTED)**:
extending the probe to secp192k1 and secp224k1 (both Koblitz curves
with near-power-of-2 group orders), via
[`tests/lll_degeneracy_probe.rs::probe_koblitz_lll_degeneracy_hypothesis`](tests/lll_degeneracy_probe.rs):

| Curve     | n bit-length | LLL outcome (2 seeds) | Time per probe |
|-----------|--------------|------------------------|----------------|
| secp192k1 |          192 | 2/2 RECOVERED          | ~320 ms        |
| secp224k1 |          224 | 2/2 RECOVERED          | ~630 ms        |
| secp256k1 |          256 | 0/2 (iteration cap)    | ~10000 ms      |
| P-256     |          256 | 2/2 RECOVERED          | ~700 ms        |

The Koblitz curves secp192k1 and secp224k1 — both having near-
power-of-2 group orders — succeed at LLL **fine**.  The
degeneracy is therefore **specific to secp256k1**, NOT a general
near-power-of-2 phenomenon.

Refined hypotheses (all currently open):

- The phenomenon may relate to a specific arithmetic feature of
  secp256k1's `n` beyond bit-length and near-power-of-2 form.
- It may correlate with the **trace** `t ≈ 2^{129}` having a
  specific structure (e.g., the CM `Z[ω]` factorisation produces
  a particularly degenerate basis).
- It could be a numerical-stability quirk of the LLL implementation
  at the boundary where dim × log(B) approaches the iteration cap;
  P-256's specific `n` produces slightly smaller intermediate
  values, secp256k1's slightly larger.

The phenomenon is real and reproducible but its **underlying
mechanism remains open** even after the cross-Koblitz probe.
This affects Phase 2 directly: the GLV-aware lattice for secp256k1
likely inherits the same degeneracy, requiring stronger reduction
(BKZ-15+) or a different lattice formulation.  See
[`RESEARCH_GLV_HNP_PHASE2.md`](RESEARCH_GLV_HNP_PHASE2.md) §4 for
the Phase 2 risk mitigation strategy.

### Phase 1: lattice prototype + planted-key recovery (toy)

- Build the GLV-aware lattice in PARI/GP using small (40-bit)
  secp256k1-like curve.
- Generate `m` ECDSA signatures with GLV decomposition and
  controlled bias on `k_{i,1}` (top bit = 0).
- Run LLL on the lattice; verify it recovers the planted `d`.
- Compare recovery rate vs. standard HNP on the same signatures.

**Predicted Phase 1 outcome**: at 40-bit curves, both attacks
recover `d`; GLV-aware uses ~2× more lattice dimension but ~half
the signatures.

### Phase 2: empirical scaling

- Run on 60-bit, 80-bit curves.  Measure:
  - Signatures-to-recovery curve
  - Wall-clock LLL reduction time
  - Comparison with Bleichenbacher's DFT for fractional bias

**Predicted Phase 2 outcome**: GLV-aware lattice scales similarly
to standard HNP up to dimension ~500 (the practical LLL limit on
2024-era hardware).  Beyond that, BKZ-30 or similar needed.

### Phase 3: real-world bias profiles

- Audit the secp256k1 implementations (libsecp256k1, OpenSSL,
  BouncyCastle) for k_1-only-leak vulnerabilities.
- For each, characterize the actual leak bits per signature.
- Estimate practical attack feasibility.

**Predicted Phase 3 outcome**: the well-audited libraries
(libsecp256k1) have no k_1-only leak; the older / hand-rolled
implementations may.  Most likely candidates: hardware wallets and
TPMs that hand-rolled GLV scalar multiplication without paying
attention to the secrecy of half-scalars separately.

### Phase 4: practical attack on a real target

- If Phase 3 finds a real leak, build the end-to-end attack:
  collect signatures, run lattice, recover key.
- Compare against published HNP attacks on the same target.

This is the speculative end of the programme — depends entirely
on whether Phase 3 surfaces a real target.

## 5. Relationship to other attack programmes

### 5.1 Orthogonal to the isogeny-graph programme

Per [`RESEARCH_HNP_LANDSCAPE.md`](RESEARCH_HNP_LANDSCAPE.md) §5,
HNP attacks attack the **scalar side** (recover `d` from biased `k`),
while isogeny-graph attacks attack the **group side** (transport
ECDLP between curves).  No interaction in either direction.

The structural-completeness theorem in
[`PAPER_STRUCTURAL_COMPLETENESS.md`](PAPER_STRUCTURAL_COMPLETENESS.md)
is unaffected by HNP advances, and conversely.

### 5.2 Connection to multi-key HNP

The existing `crypto/src/cryptanalysis/multi_key_hnp.rs` does
multi-key HNP (BBSP): stacked lattice with multiple keys signed
under shared-but-biased nonce streams.

A **multi-key GLV-aware HNP** is a natural extension — multiple
secp256k1 keys, each with k_1-bias-leak — but introduces an extra
layer of complexity that should be staged after Phase 1-3.

### 5.3 Connection to deterministic-ECDSA bypasses

If a deterministic-ECDSA implementation (RFC 6979) has the GLV
decomposition itself leak via side-channel **even when `k` is
deterministic**, then GLV-aware HNP applies.  This is a non-obvious
extension of the "deterministic ECDSA is HNP-immune" lore:
deterministic *with respect to* `k` does not mean deterministic
*with respect to* `(k_1, k_2)` if the decomposition is done in a
side-channel-leaky way.

## 6. Defenses

If the proposal validates (Phase 1-2 show practical attacks at
realistic bias levels), the defenses are:

1. **Constant-time GLV decomposition**: never leak intermediate
   half-scalar values via timing or cache patterns.
2. **Joint scalar multiplication**: use Strauss-Shamir or windowed
   joint multiplication so that `(k_1, k_2)` are processed in
   parallel without separable per-half code paths.
3. **Randomized projective coordinates**: per-multiplication
   randomization of the projective representation, scrambling
   half-scalar correlations.
4. **Fall back to non-GLV scalar multiplication** when side-channel
   resistance is paramount and the 2× speedup isn't worth the
   audit overhead.

## 7. Reproducibility roadmap

When implementations exist, they should be in:

- `crypto/src/cryptanalysis/glv_hnp.rs` — Rust prototype of the
  GLV-aware lattice attack.  Stage 1: toy-size lattice + planted-d
  recovery.  Stage 2: scale-up via PARI/fpylll bindings.
- `crypto/tests/glv_hnp_audit.rs` — integration test analogous
  to `tests/curve_audit.rs`: given a curve with GLV (currently only
  secp256k1), assert the GLV-aware lattice geometry is well-defined.

Currently neither exists.  The next concrete deliverable is the
Phase 1 PARI prototype.

## 8. Open questions

- Is the GLV-aware HNP variant **information-theoretically optimal**
  for the k_1-only leak model?  (Lower bound question.)
- Does the lattice generalize to other CM endomorphism structures
  (e.g., j=1728 with Z[i]-multiplication, or generic-CM curves with
  small-disc orders)?  These curves don't usually have GLV deployed,
  but the math generalizes.
- For curves with **two non-trivial endomorphisms** (very rare, e.g.,
  some CM-by-Z[i] curves), can a 3D-aware HNP variant exploit the
  third dimension?
- Can the GLV-aware lattice be **batched across multiple curves** —
  e.g., a single attacker observing secp256k1 + P-256 signatures
  under correlated bias — to amortize lattice-reduction cost?
